# DocCheck · AI 文档自动化合规检查工具 · 需求文档

**版本：** v1.0  
**日期：** 2026-05-25  
**设计基准：** ~100人用户 / 内网部署 / 2核2G服务器  
**许可证：** MIT  

---

## 一、系统概述

企业级 AI 文档合规检查工具。上传 Word 文档 → AI 自动对照规则检查 → 生成检查报告 → 审核确认。

### 核心流程

```
文档编写者登录
  ↓
选择文档类型 → 上传 .docx 文件
  ↓
系统自动解析文档 → 匹配该类型的启用规则
  ↓
AI（LLM）逐条检查 → 生成检查报告
  ↓
编写者查看报告 → 按建议修改文档（外部编辑）
  ↓
审核人员审核 → 确认/驳回/忽略问题项
  ↓
生成审核结论（通过 / 有条件通过 / 不通过）
```

---

## 二、用户角色

| 角色 | 人数 | 职责 |
|:----|:----|:------|
| **文档编写者** | ~90人 | 上传文档、发起检查、查看报告、导出报告 |
| **审核人员** | ~10人 | 审核检查结果，确认/驳回/忽略问题项，出审核结论 |
| **规则管理员** | 1~3人 | 管理检查规则（增删改查），管理文档类型分类 |
| **系统管理员** | 1人 | 配置 LLM 地址/Key、用户管理、系统设置 |

> 同一人可同时拥有多个角色（如：既是编写者又是审核人员）。

## 三、功能需求

### 3.1 规则管理

规则管理员通过 Web 界面管理检查规则。

#### 3.1.1 规则字段

| 字段 | 类型 | 必填 | 说明 |
|:----|:----|:----:|:-----|
| **规则名称** | 字符串 | ✅ | 简短名称，如"必须包含投资估算章节" |
| **文档类型** | 枚举 | ✅ | 关联到一种文档类型（立项报告 / 技术规格书 / …） |
| **规则描述** | 文本 | ✅ | 自然语言描述规则具体内容，供 LLM 判断 |
| **严重程度** | 枚举 | ✅ | 🔴必须改 / 🟡建议改 |
| **适用阶段** | 枚举 | 🔲 | 初审 / 终审 / 通用 — 不选则适用所有阶段 |
| **排序号** | 整数 | 🔲 | 报告中的展示顺序 |
| **启用状态** | 布尔 | ✅ | 启用 / 禁用 |

**规则描述示例：**

> "立项报告应当包含'投资估算'或'经费预算'相关章节，明确列出项目总预算、资金来源、分项费用构成，费用应与项目范围匹配。"

> "技术规格说明书应对每个功能模块给出明确的性能指标（如响应时间<2s、并发数>1000），不能只写'满足要求'这类模糊表述。"

#### 3.1.2 规则操作

| 操作 | 说明 |
|:----|:------|
| **新增规则** | 填写字段，选择文档类型和严重程度 |
| **编辑规则** | 修改规则内容 |
| **删除规则** | 删除前确认（已关联检查报告的规则不能硬删，标记为"已废弃"） |
| **启用/禁用** | 禁用的规则不参与检查 |
| **复制规则** | 从已有规则复制一份，修改后另存为新规则 |
| **批量调整** | 批量启用/禁用某文档类型下的规则 |
| **规则排序** | 拖拽排序，影响报告中的展示顺序 |

#### 3.1.3 文档类型管理

| 操作 | 说明 |
|:----|:------|
| **新增文档类型** | 如"立项报告""技术规格说明书""验收报告" |
| **编辑/删除** | 删除时确认是否有规则关联 |
| **优先级排序** | 影响上传页面的展示顺序 |

### 3.2 文档上传与检查

#### 3.2.1 上传页面

1. 用户选择「文档类型」（下拉选择，按优先级排序）
2. 选择「检查阶段」（初审 / 终审 / 通用 — 可选）
3. 上传 .docx 文件（拖拽或点击上传）
4. 点击「开始检查」按钮

**限制：**
- 单文件最大 50MB
- 仅支持 .docx 格式（后续可扩展 .doc、.pdf）
- 不支持加密/带密码的文档

#### 3.2.2 检查过程

1. 系统用 python-docx 解析文档，提取：
   - 全文文本
   - 章节结构（按 Heading 级别拆分）
   - 段落归属章节
2. 自动匹配该文档类型下 **已启用** 的所有规则
3. 如果是「初检」，匹配「初审 + 通用」阶段规则
4. 调用 LLM 进行判断（批量检查模式：全部规则 + 文档全文一次发给 LLM）
5. 生成检查报告

#### 3.2.3 LLM 调用设计

**调用方式：** 批量检查（推荐）

```
系统 Prompt：
"你是一个文档合规检查助手。以下是文档全文和需要检查的规则列表。
请逐条判断文档是否符合每条规则，输出结构化结果。"

文档全文：[文档全文内容]

规则列表：
1. 规则名称：必须包含投资估算章节
   规则描述：立项报告应当包含...
   严重程度：必须改
2. 规则名称：技术方案应有对比分析
   规则描述：技术规格说明书应对每个功能模块...
   严重程度：建议改

请输出 JSON 格式结果：
[
  {
    "rule_name": "必须包含投资估算章节",
    "compliant": false,
    "issue": "未发现投资估算相关章节",
    "location": "全文未覆盖",
    "suggestion": "建议在第4章增加投资估算小节"
  }
]
```

**LLM 配置：** 系统管理员在后台配置 API 地址和 Key（兼容 OpenAI 格式的接口）。模型切换、超时时间、重试次数均可配置。

### 3.3 检查报告

#### 3.3.1 报告页面

```
┌──────────────────────────────────────────┐
│  DocCheck 检查报告                        │
│  ─────────────────────────────            │
│  文档：XX项目立项报告_v2.1.docx            │
│  类型：立项报告  检查时间：2026-05-25 14:30│
│  检查人：张三                             │
├──────────────────────────────────────────┤
│  检查结果摘要                            │
│  ┌──────┬──────┬──────┬──────┐          │
│  │ 总规则 │ 通过  │ 问题  │ 通过率 │        │
│  │  12   │  8   │  4   │  67%  │          │
│  └──────┴──────┴──────┴──────┘          │
├──────────────────────────────────────────┤
│  问题列表                                │
│  ┌─┬────────┬────────┬────┬────┬──────┐ │
│  │#│规则名称 │问题描述 │位置│严重 │建议  │ │
│  │1│投资估算 │未包含… │全文 │🔴  │建议… │ │
│  │2│性能指标 │缺少… │3.2节│🟡  │建议… │ │
│  │3│封面信息 │完整    │封面│🟢  │—    │ │
│  └─┴────────┴────────┴────┴────┴──────┘ │
├──────────────────────────────────────────┤
│  操作区                                  │
│  [导出PDF] [导出Word] [重新检查]         │
└──────────────────────────────────────────┘
```

#### 3.3.2 报告数据模型

每条检查结果包含：

| 字段 | 说明 |
|:----|:------|
| **规则名称** | 对应的规则 |
| **是否合规** | 是 / 否 / 无法判断 |
| **问题描述** | AI 判断的具体问题 |
| **原文位置** | 问题所在的章节或段落 |
| **原文引用** | 问题部分的原文摘要 |
| **严重程度** | 必须改 / 建议改 / 通过 / 无法判断 |
| **修改建议** | AI 给出的修改建议 |
| **审核状态** | 待审核 / 已确认 / 已驳回 / 已忽略 |

#### 3.3.3 报告操作

| 操作 | 说明 |
|:----|:------|
| **查看** | 浏览器直接查看完整报告 |
| **导出 PDF** | 导出为 PDF 文件 |
| **导出 Word** | 导出为 .docx 文件 |
| **重新检查** | 上传修改后的文档再次检查，保留历史版本 |
| **分享** | 生成报告链接供他人查看（权限校验） |

### 3.4 审核流程

审核人员进入报告后，逐条审核 AI 的判断结果。

#### 3.4.1 审核操作

| 操作 | 说明 | 图标 |
|:----|:------|:----:|
| **确认** | 认可 AI 的判断，问题确实存在 | ✅ |
| **驳回** | AI 误判，该条没有问题 | ❌ |
| **忽略** | 问题存在但本次可接受，不做强制修改 | ⏭️ |
| **备注** | 对审核结果添加备注说明（可选） | ✏️ |

#### 3.4.2 审核结论

审核人员审核完所有问题后，出具审核结论：

| 结论 | 条件 | 说明 |
|:----|:-----|:------|
| **✅ 通过** | 无"必须改"级别的问题 | 文档符合规范 |
| **⚠️ 有条件通过** | 存在"必须改"问题但已被确认 | 需修改后重新检查确认 |
| **❌ 不通过** | 存在"必须改"问题且未解决 | 文档需重大修改 |

#### 3.4.3 审核记录

- 每条问题的审核操作记入审计日志
- 记录操作人、操作时间、操作类型、备注

### 3.5 历史记录

#### 3.5.1 文档历史

用户查看某份文档的历次检查记录：

```
文档：XX项目立项报告_v2.1.docx
├─ 第3次检查 2026-05-25 14:30 → 12规则 / 4问题
├─ 第2次检查 2026-05-24 10:00 → 12规则 / 6问题
└─ 第1次检查 2026-05-23 16:00 → 10规则 / 8问题（未审核）
```

#### 3.5.2 版本对比

- 点击两次检查记录，并排对比检查结果变化
- 显示新增问题、已解决问题、仍存在的问题

### 3.6 系统管理

#### 3.6.1 LLM 配置

| 配置项 | 说明 |
|:------|:------|
| **API 地址** | LLM 服务 URL（如 `http://192.168.1.100:8000/v1`） |
| **API Key** | 认证密钥 |
| **模型名称** | 如 `qwen2.5-72b`、`gpt-4o` |
| **超时时间** | 默认 60 秒 |
| **最大重试** | 默认 3 次 |
| **温度** | 默认 0.1（低温度保证输出一致性）|
| **最大 Token** | 默认 4096 |
| **测试连接** | 点击测试能否正常调用 |

#### 3.6.2 用户管理

| 功能 | 说明 |
|:----|:------|
| **用户列表** | 查看/搜索所有用户 |
| **添加用户** | 手动添加或批量导入 |
| **编辑用户** | 修改姓名、角色、邮箱 |
| **禁用/启用** | 禁用后用户无法登录 |
| **重置密码** | 管理员可重置用户密码 |

#### 3.6.3 审核日志

- 记录所有关键操作：规则新增/修改/删除、文档检查、审核操作、系统配置变更
- 支持按时间、操作人、操作类型筛选

## 四、非功能需求

### 4.1 性能

| 指标 | 要求 |
|:----|:------|
| **并发用户** | 同时在线 ≤30人 |
| **单文档检查时间** | ≤60秒（含 LLM 调用时间） |
| **页面加载时间** | ≤2秒 |
| **报告导出时间** | ≤10秒 |
| **文件上传大小** | 单文件 ≤50MB |

### 4.2 可用性

| 指标 | 要求 |
|:----|:------|
| 系统可用性 | 99%（内网，非关键系统） |
| 故障恢复 | 重启服务即可恢复（无状态） |
| 数据备份 | SQLite 文件每日定时备份 |

### 4.3 安全

| 项目 | 要求 |
|:----|:------|
| **认证登录** | 用户名+密码，session 管理 |
| **权限控制** | 角色鉴权（编写者 / 审核 / 规则管理员 / 系统管理员）|
| **文件安全** | 上传的文档限制 .docx 格式，防恶意文件 |
| **HTTPS** | 内网 HTTPS（自签名证书） |
| **LLM API Key** | 加密存储，不在前端暴露 |

### 4.4 存储

| 数据 | 存储方式 | 预估量 |
|:----|:---------|:------|
| **检查规则** | SQLite | ≤500条 |
| **上传文档** | 文件系统 | 每个 ≤50MB，长期保留 |
| **检查报告** | SQLite + 文件 | 每年约数千份 |
| **SQLite 数据库** | 单文件 | 预计 <1GB/年 |

### 4.5 部署

| 项目 | 要求 |
|:----|:------|
| **服务器** | Linux（CentOS 7+ / Ubuntu 20.04+） |
| **最低配置** | 2核2G |
| **Python 版本** | 3.10+ |
| **依赖安装** | `pip install` 直接安装 |
| **运行方式** | systemd 托管 Uvicorn 进程 |
