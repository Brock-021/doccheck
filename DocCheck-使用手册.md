# DocCheck · 使用手册

**版本**: 2.1.0 | **更新时间**: 2026-05-28

**DocCheck** 是一款基于 AI 的文档合规检查工具。用户上传 Word 文档（.docx），系统自动调用大语言模型（LLM）进行规则检查，生成检查报告，支持审核闭环、统计分析看板、存储管理等完整功能。

## 1. 快速上手

### 1.1 登录系统

浏览器访问系统地址（如 `http://8.133.198.245:8000`），进入登录页面。

**初始账号：**

| 角色 | 用户名 | 密码 | 说明 |
|:----|:------|:----|:-----|
| 管理员 | admin | admin123 | 拥有全部权限：规则、用户、LLM 配置、审计日志、统计分析、存储管理 |
| 审核员 | reviewer | review123 | 审核所有文档检查结果、导出报告、提交审核结论 |
| 编写员 | writer | writer123 | 上传文档、查看报告 |
| 普通用户 | zhangsan | zhangsan | 上传文档、查看报告 |

> ⚠️ **首次登录请修改密码**。修改方式：登录后点击右上角用户头像 → 修改密码。

### 1.2 完整工作流程

```
┌─────────────────────────────────────────────────────────┐
│ 1. 编写者 上传文档 → 系统自动 AI 检查                     │
│ 2. 编写者 查看检查报告 → 根据问题逐条修改                  │
│ 3. 审核员 审核检查结果 → 确认/驳回/忽略 → 提交审核结论     │
│ 4. 编写者 查看最终结论 → 文档定稿                         │
│ 5. 管理员 统计分析 → 存储管理 → 系统运维                   │
└─────────────────────────────────────────────────────────┘
```

### 1.3 三步完成一次文档检查

```
第1步：上传文档 → 第2步：等待检查 → 第3步：查看报告
```

操作路径：导航栏 **上传检查** → 选择文档类型和检查阶段 → 上传 .docx 文件 → 系统自动检查 → 上传成功页面自动显示等待动画和「查看报告」按钮。

## 2. 角色说明

| 角色 | 权限 | 典型用户 |
|:----|:-----|:---------|
| **编写者** (user) | 上传文档、查看自己文档的检查报告和审核结论、查看历史版本对比 | 文档编写人员 |
| **审核员** (reviewer) | 查看所有文档报告、审核检查结果（确认/驳回/忽略）、提交审核结论、导出 PDF/Word 报告 | 文档质量管理人员 |
| **管理员** (admin) | 规则管理、文档类型管理、用户管理、LLM 配置、审计日志、统计分析看板、存储管理、全部审核权限 | 系统运维/管理员 |

> **角色叠加**：一个用户可以同时有多个角色（如 `admin,reviewer`），在用户管理中编辑角色时用逗号分隔。

## 3. 文档编写者操作

### 3.1 上传文档检查

1. 登录系统，点击 **上传检查**
2. 选择 **文档类型**（管理制度 / 技术方案 / 操作手册）
3. 选择 **检查阶段**（初稿 / 终稿）
   - **初稿**：检查格式规范、编号、术语一致性等基础要求
   - **终稿**：额外检查标点符号、页眉页脚、安全考量、异常处理等质量要求
4. 拖拽或点击选择 .docx 文档（限制最大 50MB）
5. 点击 **开始检查**
6. 页面自动跳转到检查进度页面，展示等待动画和轮播提示文字：
   - 解析文档...
   - 比对规则...
   - AI 判定...
7. 检查完成后，页面显示 **查看报告** 和 **文档列表** 两个按钮

> **提示**：检查过程通常 5~30 秒，取决于文档大小和规则数量。即使关闭页面，检查也会后台继续完成。

### 3.2 工作台首页

登录后默认进入 **工作台**，展示：

| 功能区域 | 内容 |
|:--------|:-----|
| 统计卡片 | 我的文档数、待审核报告数（仅审核员/管理员可见）、活跃规则数 |
| 最近文档 | 最近 10 个文档的最新检查状态和审核结论 |
| 快捷操作 | 点击文档名称可查看报告，点击「上传检查」可以上传新文档 |

### 3.3 查看检查报告

在 **工作台** 或 **文档列表** 中找到已检查的文档，点击 **查看报告**。

报告包含：

| 区域 | 内容 |
|:----|:-----|
| **文档信息** | 文件名、文档类型、上传者、上传时间 |
| **任务信息** | 检查阶段、适用规则数、检查状态、耗时 |
| **摘要** | 总规则数、通过数、未通过数、通过率 |
| **逐条检查结果** | 每条规则的检查结论及详情 |

**检查结果状态标识：**

| 图标 | 含义 | 说明 |
|:----|:-----|:-----|
| ✅ **通过** | 文档符合该规则 | 无需处理 |
| ❌ **不通过** | 文档存在问题 | 展开查看具体问题描述和修改建议 |
| ❓ **无法判断** | LLM 无法确认 | 需人工审核判断 |
| 🟡 **待审核** | 审核状态 | 审核员还未处理 |
| ✅ **已确认** | 审核确认 | 问题已确认，需修改 |
| ❌ **已驳回** | 审核驳回 | 驳回的认为该问题不成立，查看备注 |
| ⚪ **已忽略** | 审核忽略 | 问题可接受，不必修改，查看备注 |
| 📝 **审核结论** | 最终判定 | 通过 / 需修改 / 不通过 |

### 3.4 检查结果详情展示

展开一条检查结果，可查看：

- **问题描述**：LLM 分析的具体问题说明
- **原文位置**：问题出现在文档中的位置（章节、段落）
- **原文内容**：存在问题的原文片段
- **修改建议**：LLM 生成的修改建议

### 3.5 查看审核结论

审核员审核后，编写者在报告中可看到：

- **审核结论**：通过 / 需修改 / 不通过
- **结论备注**：审核员补充的说明
- **审核时间**：审核结论提交时间

### 3.6 查看检查历史

1. 进入 **文档列表**
2. 点击文档右侧的 **历史** 按钮
3. 查看该文档所有历史检查记录列表，包括：
   - 每次检查的时间、阶段
   - 检查状态（pending / running / done / failed）
   - 关联的报告和审核结论
4. 点击某次历史记录可跳转到对应的报告页面

### 3.7 历史版本对比

对于多次上传检查的文档，可进入 **历史对比** 页面，并排对比不同版本的检查结果变化：

| 对比项 | 说明 |
|:------|:-----|
| **新增问题** | 新版本新增了哪些未通过项 |
| **已解决问题** | 旧版本有问题的项在新版本已通过 |
| **持续问题** | 两个版本都未通过的项目 |
| **通过率变化** | 通过率的升降趋势 |

> 进入方式：在历史页面点击「版本对比」按钮，选择两个历史版本进行对比。

### 3.8 导出报告

在报告详情页，点击 **导出** 按钮：

| 格式 | 用途 | 注意 |
|:----|:----|:-----|
| **PDF** | 适合分享给外部或打印 | 需服务器安装中文字体（`fonts-noto-cjk`） |
| **Word** | 适合内部流转 | 包含完整检查结果和审核信息 |

## 4. 审核人员操作

### 4.1 查看待审核报告

1. 登录系统，进入 **工作台**
2. 顶部统计卡片显示 **待审核报告数**（统计至少有一条 pending 检查结果的报告数）
3. 点击「待审核」区域或搜索待审核的文档

> 管理员和审核员可以看到所有待审核的报告，编写者只能看到自己文档的。

### 4.2 逐条审核检查结果

1. 点击进入报告
2. 在报告页面底部，逐条查看每条检查结果
3. 对每条结果执行审核操作：

| 操作 | 含义 | 备注要求 | 操作位置 |
|:----|:-----|:---------|:---------|
| **确认** | 确认该问题存在，需修改 | 可选（自动生成确认信息） | 每条结果右侧的「确认」按钮 |
| **驳回** | 认为该问题不成立 | **必填**（说明驳回理由） | 每条结果右侧的「驳回」按钮 |
| **忽略** | 问题存在但可接受，不改 | **必填**（说明忽略原因） | 每条结果右侧的「忽略」按钮 |

### 4.3 一键全部确认

审核页面上方有一个 **一键全部确认** 按钮：

- 将当前报告中所有待审核（pending）的检查结果一键标记为「已确认」
- **不可撤销**，操作前请确认
- 适合文档问题明显、无需逐条审核的场景

### 4.4 提交审核结论

逐条处理完成后（或使用一键确认后），在报告页面提交 **审核结论**：

| 结论 | 含义 | 下一步 |
|:----|:-----|:-------|
| **通过** | 文档合规，无需修改 | 文档定稿 |
| **需修改** | 存在问题，文档需修改后重新检查 | 编写者修改后重新上传 |
| **不通过** | 文档严重不合格 | 编写者重写 |

提交结论需要填写结论备注（说明审核依据）。

### 4.5 审核状态流转

```
                ┌─────────┐
                │  待审核  │
                └────┬────┘
                     │
          ┌──────────┼──────────┐
          ▼          ▼          ▼
       ┌──────┐  ┌──────┐  ┌──────┐
       │ 确认 │  │ 驳回 │  │ 忽略 │
       └──────┘  └──────┘  └──────┘
          │          │          │
          └──────────┴──────────┘
                     │
                     ▼
              ┌────────────┐
              │ 提交结论    │
              └────────────┘
         ┌───────┼───────┐
         ▼       ▼       ▼
       ┌────┐ ┌────┐ ┌────┐
       │通过│ │需修│ │不通│
       └────┘ │改  │ │过  │
               └────┘ └────┘
```

### 5.2 规则管理

**进入方式**：管理 → 规则管理

#### 规则列表

- 按文档类型筛选规则（支持下拉筛选）
- 显示每条规则的名称、描述、严重程度、阶段、启用状态
- 搜索功能支持按规则名称快速查找

#### 新增规则

点击 **新增规则**，填写：

| 字段 | 说明 | 示例 |
|:----|:-----|:-----|
| 名称 | 规则简短名称 | 格式规范 |
| 描述 | 使用自然语言描述规则 | 文档标题应使用一级标题（Heading 1），各级子标题应使用正确的层级结构。 |
| 严重程度 | `must_fix`（必须改）/ `suggest`（建议改） | must_fix |
| 阶段 | `initial`（初稿检查）/ `final`（终稿检查）/ `all`（全部阶段） | all |
| 所属类型 | 适用于哪种文档类型（**可多选**） | 管理制度, 技术方案 |

> **规则描述不限制格式**：纯自然语言即可，无需写代码或正则表达式。LLM 会自动理解规则含义并判断文档是否符合。
> **文档类型可多选**：一条规则可以同时适用于多种文档类型，新增/编辑时通过多选框勾选即可。

#### 编辑规则

点击规则右侧的编辑按钮，修改规则信息后保存。

#### 启用/禁用规则

点击规则右侧的开关按钮：

- **启用**（绿色）：规则将在下次文档检查时生效
- **禁用**（灰色）：规则暂时不参与检查，但保留配置

#### 删除规则

点击规则右侧的删除按钮。删除后不可恢复，请谨慎操作。

### 5.3 文档类型管理

**进入方式**：管理 → 文档类型管理

管理员可以：

| 操作 | 说明 |
|:----|:-----|
| **新增类型** | 创建新的文档类型（如"验收文档""合同模板"） |
| **编辑类型** | 修改类型名称和排序，同时可 **勾选/取消** 该类型适用的检查规则 |
| **删除类型** | 删除不需要的类型（需确认无关联规则和文档） |

**预置的文档类型：**

| 类型 | 用途 | 规则数 |
|:----|:-----|:-------|
| 管理制度 | 公司规章制度类文档 | 6 条 |
| 技术方案 | 技术设计/实施方案 | 3 条 |
| 操作手册 | 操作指南/用户手册 | 3 条 |

> 新增文档类型后，需要为该类型绑定检查规则，该类型的文档上传时才会执行检查。在编辑文档类型时也可勾选/取消关联的规则。

### 5.4 用户管理

**进入方式**：管理 → 用户管理

#### 用户列表

- 显示所有用户：用户名、显示名、角色、状态（启用/禁用）、最近登录时间
- 按条件筛选和搜索

#### 新增用户

点击 **新增用户**，填写：

| 字段 | 说明 | 示例 |
|:----|:-----|:-----|
| 用户名 | 登录账号，唯一不可重复 | wangwu |
| 显示名 | 界面显示的名称 | 王五 |
| 密码 | 初始密码 | user123456 |
| 角色 | `writer`(编写者) / `reviewer`(审核员) / `admin`(管理员) | writer |

> 角色支持组合，如 `writer,reviewer` 表示既是编写者又是审核员。

#### 编辑用户

| 操作 | 说明 |
|:----|:-----|
| 修改显示名 | 更新用户显示名称 |
| 修改角色 | 变更用户权限 |
| 重置密码 | 生成新密码（系统自动生成随机密码，需复制给用户） |

#### 禁用/启用用户

- **禁用**：用户无法登录系统，但数据保留
- **启用**：恢复用户登录权限

### 5.5 LLM 配置

**进入方式**：管理 → LLM 配置

配置 AI 检查引擎的模型参数。详见部署手册第 4 节。

**配置项：**

| 配置项 | 说明 | 注意事项 |
|:------|:-----|:---------|
| API Base URL | LLM 服务的 API 地址 | 内网部署填写内网地址 |
| API Key | 认证密钥 | 不会明文显示在页面上 |
| Model | 模型名称 | 支持切换不同模型 |
| Timeout | 单次超时（秒） | 文档较大时建议 120s+ |
| Temperature | 生成温度（0.0~1.0） | 规则检查建议 0.1~0.3 |

> 配置保存后立即生效，无需重启服务。

### 5.6 审计日志

**进入方式**：管理 → 审计日志

记录所有关键操作，供审计追溯。

**支持的审计事件类型：**

| 操作类型 | 触发场景 |
|:--------|:---------|
| `login` | 用户登录 |
| `logout` | 用户登出 |
| `document_upload` | 上传文档 |
| `rule_create` | 新增规则 |
| `rule_update` | 修改规则 |
| `rule_delete` | 删除规则 |
| `user_create` | 新增用户 |
| `user_update` | 修改用户 |
| `user_disable` | 禁用用户 |
| `user_enable` | 启用用户 |
| `password_reset` | 重置密码 |
| `llm_config` | LLM 配置变更 |
| `review_action` | 审核操作 |
| `conclusion` | 提交审核结论 |
| `storage_cleanup` | 存储清理 |

**查询过滤：**

- 按操作类型筛选
- 按时间范围筛选（开始日期 ~ 结束日期）
- 日志列表：时间、操作用户、操作类型、操作详情、IP 地址

> 审计日志为 append-only，不会被清理或删除（存储清理功能只删除文档关联数据，不影响审计日志）。

### 5.7 统计分析看板

**进入方式**：管理 → 统计分析

提供全面的数据可视化看板，支持时间范围筛选。

**操作：**

1. 进入统计分析页面
2. 可选：设置时间范围（开始日期 ~ 结束日期），筛选指定时间段的数据
3. 点击 **查询** 更新数据
4. 查看各统计模块的数据

#### 概览统计

| 指标 | 说明 |
|:----|:-----|
| 📄 总文档数 | 系统所有上传的文档数量（按时间筛选则为该时间段上传量） |
| 🔄 总检查次数 | 系统总执行检查次数 |
| ✅ 检查成功 | 状态为 done 的检查次数 |
| ❌ 检查失败 | 状态为 failed 的检查次数 |
| 👥 用户数 | 系统注册用户总数 |

#### 按文档类型分布

展示每种文档类型的文档数和检查次数，帮助了解不同类型文档的使用情况。

#### 用户活跃度 Top10

按文档上传数排序的前 10 名用户，显示每个用户的上传文档数和检查次数。

#### 合规统计

| 指标 | 说明 |
|:----|:-----|
| 通过数 | compliant = true 的检查结果数 |
| 未通过数 | compliant = false 的检查结果数 |
| 无法判断数 | compliant = null 的检查结果数 |
| 通过率 | 通过数 / 总数 × 100% |
| 未通过率 | 未通过数 / 总数 × 100% |

#### 审核状态统计

| 状态 | 说明 |
|:----|:-----|
| 待审核 (pending) | 尚未处理的检查结果数 |
| 已确认 (confirmed) | 审核员确认问题存在的检查结果数 |
| 已驳回 (rejected) | 审核员驳回的检查结果数 |
| 已忽略 (ignored) | 审核员忽略的检查结果数 |

#### 报告结论分布

| 结论 | 说明 |
|:----|:-----|
| 通过 (pass) | 文档合规通过 |
| 条件通过 (conditional_pass) | 有条件通过 |
| 不通过 (fail) | 文档审核未通过 |
| 待定 (pending) | 尚未给出结论 |

#### 每日检查趋势

近 30 天（或指定时间范围内）每日检查次数的变化趋势图，用于监控系统使用量变化。

#### 规则命中排行 Top20

展示哪些规则命中次数最多、导致失败最多的规则 Top20，帮助管理员优化规则配置。例如，某规则频繁触发可能是文档共性问题，需要加强培训或修改模板。

#### 视图切换

统计分析页面支持四种视图切换：

| 视图 | 说明 |
|:----|:-----|
| 📊 柱状图 | 适合展示各分类维度对比 |
| 🥧 饼图 | 适合展示占比分布 |
| 📈 趋势图 | 适合展示时间变化趋势 |
| 📋 表格 | 展示精确数值数据，可复制 |

### 5.8 存储管理

**进入方式**：管理 → 存储管理

用于管理磁盘空间，清理旧文档及其关联数据。

#### 存储概览

管理员进入存储管理页面后，查看当前存储状况：

| 指标 | 说明 |
|:----|:-----|
| 文档记录数 | 数据库中文档记录总数 |
| 存储文件数 | `uploads/` 目录中的文件总数 |
| 已用空间 | `uploads/` 目录占用的磁盘大小 |

#### 清理旧文档

显示四个时间段分别有多少个文档可清理：

| 时间段 | 预期可清理量 |
|:------|:-----------|
| 30天前 | 例：42 个文档 |
| 60天前 | 例：28 个文档 |
| 90天前 | 例：15 个文档 |
| 180天前 | 例：5 个文档 |

点击对应时间段的 **清理** 按钮：

1. 弹出确认弹窗，显示将删除的文档数和预计释放空间
2. 点击确认后执行清理
3. 清理过程中显示进度条和状态文字
4. 清理完成后显示结果：

```
已清理 15 个文档，释放 45.2 MB
清理时间截止：2026-02-24T14:30:00（90天前）
```

#### 清理影响

| 影响范围 | 说明 |
|:--------|:-----|
| ✅ 解放磁盘空间 | 删除物理文件和数据库记录 |
| ✅ 不影响用户账号 | 用户、角色、权限不受影响 |
| ✅ 不影响规则和文档类型 | 检查规则、文档类型配置保留 |
| ✅ 不影响审计日志 | 审计操作记录不会被删除 |
| ❌ 不可恢复 | 删除后无法恢复文档和检查记录 |

> **注意事项**：
> - 清理操作不可撤销，请确认后再执行
> - 建议备份数据库后再执行大规模清理
> - 大文档较多时清理可能需要数秒到数十秒

## 6. 常见场景

### 场景一：编写管理制度文档

```
1. 管理员创建"管理制度"类型的检查规则
2. 编写者登录 → 上传检查 → 选择"管理制度" → 上传 .docx
3. 系统自动检查 → 生成报告
4. 编写者查看报告，根据问题逐条修改
5. 重新上传 → 再次检查 → 直至所有问题通过
6. 审核员审核报告 → 确认/驳回/忽略 → 提交结论"通过"
7. 编写者查看最终结论 → 文档定稿
```

### 场景二：审核人员批量审核

```
1. 审核员登录 → 工作台 → 查看待审核报告数（42份）
2. 点击第1份报告 → 符合预期 → 点击"一键全部确认"
3. 提交审核结论"通过" → 填写备注
4. 返回工作台 → 点击第2份报告...
```

### 场景三：管理员定期维护

```
1. 登录 → 管理 → 统计分析 → 查看近30天趋势 → 导出分析结果
2. 管理 → 规则管理 → 查看 Top20 规则命中 → 调整规则
3. 管理 → 存储管理 → 清理90天前的旧文档
4. 管理 → 审计日志 → 查看近期操作记录
```

### 场景四：用户上传多个版本迭代

```
1. 第1次上传：初稿 → 检查出5个问题
2. 修改后第2次上传：初稿 → 检查出2个问题（已解决3个）
3. 提交终稿阶段 → 上传 → 选择"终稿" → 检查出标点符号问题
4. 修改后第3次上传：终稿 → 全部通过
5. 查看历史对比 → 看到从5个→2个→0个的改进过程
```

### 场景五：磁盘空间告急处理

```
1. 发现磁盘告警 → 管理员登录
2. 管理 → 存储管理 → 查看存储概览
3. 显示：文档记录 5000个，已用空间 2.3GB
4. 选择"90天前" → 预览：1200个文档可清理
5. 点击清理 → 确认 → 等待清理完成
6. 释放空间约 600MB
```

## 7. FAQ

### Q: 支持哪些文档格式？
目前仅支持 `.docx` 格式（Word 文档）。单个文件最大 50MB。

### Q: 检查规则支持哪些内容？
规则使用自然语言描述，LLM 会自动理解并判断。例如：
- "文档标题应使用一级标题"
- "正文应使用全角中文标点"
- "技术方案应包含系统架构描述"

### Q: 检查速度如何？
通常 5~30 秒完成一次检查，取决于文档长度和规则数量。大型文档（50页以上）可能需要 60~90 秒。

### Q: 检查结果准确吗？
LLM 检查为辅助工具，建议所有检查结果由审核员人工复核后再做最终结论。

### Q: 可以自定义检查规则吗？
可以。管理员在规则管理页面，用自然语言新增规则即可，无需技术背景。（推荐先用现有规则运行观察，再根据实际需要补充自定义规则）

### Q: 如何查看谁上传了文档？
文档列表页和报告详情页都会显示上传者的姓名和上传时间。

### Q: 文档会保存多久？
文档和检查记录永久保存，除非管理员在存储管理页面手动清理。建议根据业务需求定期清理（如每季度清理一次 90 天前的旧文档）。

### Q: 检查时断网了怎么办？
检查过程由后台异步执行（FastAPI + uvicorn），即使页面关闭，检查也会继续。重新打开报告页面即可查看结果。但如果 LLM API 不可用，检查任务会失败，可在 LLM 配置页检查 API 状态后重试。

### Q: 统计分析数据多久更新一次？
统计数据实时查询数据库，每次打开页面或点击查询按钮时都会获取最新数据。

### Q: 支持切换 AI 模型吗？
支持。管理员在 LLM 配置页修改 Model 字段即可，无需重启服务。支持 DeepSeek、OpenAI、通义千问等兼容 OpenAI API 格式的模型。

### Q: 忘记密码了怎么办？
联系管理员重置密码。如果是管理员自己忘记了密码，按部署手册第 9 节的方法直接在数据库重置。

### Q: 系统最多支持多少用户和文档？
理论无上限（SQLite 支持 TB 级数据），但推荐：
- 日检查量 500~1000 次：运行流畅
- 日检查量 1000 次以上：建议迁移到 MySQL/PostgreSQL

如遇系统性能问题，可联系管理员优化配置或升级服务器。
