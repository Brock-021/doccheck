**版本：** v0.2（方案阶段）  
**日期：** 2026-05-28  
**项目名称：** DocCheck  
**许可证：** MIT  

---

## 一、项目定位

基于 AI 的轻量级文档合规检查工具。上传 Word 文档 → AI 自动对照规则检查 → 生成检查报告。

> 不做重平台，不做工作流引擎，不做复杂的多人协同——做好"上传→检查→出报告"这一核心闭环。

---

## 二、选型决策

| 层面 | 选型 | 理由 |
|:----|:-----|:------|
| **Web 框架** | **FastAPI**（Python） | 现代异步，自带 OpenAPI 文档，性能好，100人场景绰绰有余 |
| **数据库** | **SQLite** | 零安装零运维，单文件数据库，100人规模完全够用 |
| **前端** | **Jinja2 模板 + HTML** | 不搞前后端分离，FastAPI 原生渲染，开发效率最高 |
| **文档解析** | **python-docx** | .docx 格式解析，提取文本+段落+章节结构 |
| **LLM 调用** | **httpx / openai SDK** | 调用内网 LLM API，兼容 OpenAI 格式 |
| **部署方式** | **systemd + Uvicorn** | 单进程部署，2核2G服务器足够 |

## 四、核心功能

### 4.1 规则管理

规则以「自然语言描述」存储，交给 LLM 判断文档是否符合。

**一条规则的构成：**

```
文档类型：立项报告, 技术方案（可关联多个）
规则名称：必须包含投资估算章节
规则描述：立项报告应当包含"投资估算"或"经费预算"相关章节，明确列出项目总预算、资金来源、分项费用构成。
严重程度：必须改
适用阶段：初审
启用状态：✅ 启用
```

> 规则就是一段描述文字，业务人员直接在网页上写，不用写代码，不用写正则。LLM 会理解语义来判断。

规则可关联 **多种文档类型**（多对多），支持按文档类型筛选、启用/禁用。新增规则时通过多选框选择适用的文档类型。在文档类型编辑页也可勾选/取消关联的规则，实现双向管理。

### 4.2 文档上传与检查

1. 用户选择「文档类型」（立项报告 / 技术规格书 / 需求文档 / …）
2. 上传 .docx 文件
3. 系统自动提取文档全文 + 章节结构
4. 系统自动匹配该文档类型下启用的规则
5. 逐条提交给 LLM 判断，或一次性提交全部规则
6. 生成检查报告

**检查方式（两种模式）：**

| 模式 | 说明 | 适用场景 |
|:----|:------|:---------|
| **逐条检查** | 每条规则独立调用 LLM，结果精确，可定位到问题段落 | 规则少（<30条）、需要精确定位 |
| **批量检查** | 所有规则+文档全文一次发给 LLM，一次出全部结果 | 规则多、效率优先 |

推荐先用「批量检查」模式，后续优化再加逐条模式。

### 4.3 检查报告

报告包含以下内容：

- **基本信息**：文档名称、文档类型、检查时间、检查人
- **检查结果摘要**：总规则数、通过数、问题数、通过率
- **问题列表**（逐条）：

| 序号 | 规则名称 | 问题描述 | 原文位置 | 严重程度 | 修改建议 | 状态 |
|:----:|:---------|:---------|:---------|:--------|:---------|:----:|
| 1 | 必须包含投资估算章节 | 未发现投资估算相关章节 | 全文 | 🔴 必须改 | 建议在第4章增加投资估算小节，列出项目总预算及分项费用 | 待审核 |
| 2 | 技术方案应有对比分析 | 提到了技术方案但未做对比 | 第3.2节 | 🟡 建议改 | 建议补充至少2种技术方案的对比表（优劣势/成本/风险） | 待审核 |
| 3 | 封面信息完整 | 封面包含项目名称、编制单位、日期 | 封面 | 🟢 通过 | — | 已通过 |

**严重程度分级：**
- 🔴 **必须改** — 不满足硬性要求，审核不通过
- 🟡 **建议改** — 缺少推荐内容，但不影响基本合规
- 🟢 **通过** — 完全符合规则
- ⚪ **无法判断** — AI 不确定，需人工判断

**报告导出：** 支持导出 PDF / Word 格式

### 4.4 审核流程

审核人员对每条问题做判定：

| 操作 | 含义 |
|:----|:------|
| ✅ **确认** | 认可 AI 的判断，问题确实存在 |
| ❌ **驳回** | AI 误判，该条没有问题 |
| ⏭️ **忽略** | 问题存在但本次可接受，不做强制修改 |

审核后可一键生成「审核结论」（通过 / 有条件通过 / 不通过）。

### 4.5 历史记录

- 同一文档多次检查，按版本保留
- 列表可查看每次检查的时间、结果摘要
- 支持查看某次检查的完整报告

## 六、2核2G 适配说明

| 组件 | 资源占用 | 说明 |
|:----|:---------|:-----|
| FastAPI + Uvicorn | ~50MB | 单进程，内存占用极低 |
| SQLite | ~5MB | 文件型数据库，几乎不占内存 |
| 文件解析（python-docx） | 临时 ~100MB | 只在上传时触发，解析完后释放 |
| LLM 调用 | 无内存占用 | 只发 HTTP 请求等待响应 |

**结论：** 2核2G 跑 DocCheck **完全足够**，留出约 1.5GB 空闲内存。

## 七、部署方式

```
# 安装依赖
pip install fastapi uvicorn python-docx sqlalchemy jinja2 aiofiles

# 启动
uvicorn main:app --host 0.0.0.0 --port 8000
```

生产环境用 systemd 托管：
```
[Service]
ExecStart=/usr/bin/uvicorn main:app --host 0.0.0.0 --port 8000
WorkingDirectory=/opt/doccheck
Restart=always
User=doccheck
```
